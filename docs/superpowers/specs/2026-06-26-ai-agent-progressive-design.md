# AI Agent 渐进式构建设计

## 概述

基于"AI Oncall Agent 实战总结"的思路，为中等规模团队（10-50 人）设计一套渐进式 AI Agent 构建方法论。核心策略：先极简原型验证价值，再逐步演进为多角色、平台化架构。

## 三阶段演进路线

### 阶段一：极简单 Agent（第 1-3 天）

目标：快速跑通，验证 AI Agent 在你的场景中真的能解决问题。

```
IM 平台 → 消息目录（文件订阅）→ CLI Harness（长驻）→ 知识库 + 代码仓库 → tmux 托管
```

- 1 个 Agent + 1 份知识库 + prompt 驱动 + tmux 托管
- 通过 IM 平台的 CLI 工具订阅消息（如 `lark-cli event subscribe`）
- Agent 长驻消费消息目录，不对每条消息重建上下文
- 核心依赖：可信知识库，而非 RAG

### 阶段二：多角色 + 流水线化（第 1-3 周）

升级信号：上下文开始混杂、多用户并发时回答质量下降、需要第二个角色分担。

新增：
- **消息流水线抽象：** 订阅→通知→检测→唤醒→批量处理（详见下文）
- **三个 Agent 角色：** Oncall Agent / 监工 Agent / 代码分析 Agent
- **监工异常告警：** 发现异常自动通知

### 阶段三：平台化（第 4 周起，按需演进）

升级信号：用户量持续增长、单实例扛不住、需要工单追溯和数据分析。

新增：
- Webhook → MQ → 按用户 sharding
- 多 Agent 实例独立消费
- 上下文持久化到数据库，形成工单
- MCP 加载工单上下文
- 定时优化闭环

## 消息消费流水线

五个环节各司其职：

```
订阅 → 通知 → 检测 → 唤醒 → 批量处理
```

### 职责边界

**外部脚本负责（不消耗 Agent 上下文）：**
- 订阅：监控 IM 消息目录
- 通知：感知新消息到来
- 检测：判断 Agent 是否忙碌，是否有人工 attach 到 tmux。AI 判断 + 规则兜底
- 唤醒：通过 tmux send-keys 向 Agent session 发送唤醒指令，同时注入"回忆关键原则"prompt snippet

**AI Agent 负责：**
- 批量处理：一次性读取积压消息，关联同一用户的多轮上下文
- 归档：写入 worklog
- 升级：需要时交给其他角色

### 批量处理优于逐条处理

逐条处理响应虽快，但丢失用户多轮上下文。批量处理让 Agent 能看到同一用户的多条消息，避免上下文断裂。

## 上下文工程

Agent 回答质量的核心不是模型，而是它"看到了什么"。

### 三层上下文金字塔

1. **Prompt 层（最小、最核心）：** CLAUDE.md/AGENTS.md，需持续精简
2. **知识库层（按需加载）：** 可信文档 + 代码索引
3. **代码仓库层（按需浏览）：** 直接读源码，不通过 RAG
4. **Worklog 层（历史上下文）：** 每日记录，跨天压缩

### 知识库四原则

1. **可信优先：** 所有内容人工校对确认，宁可缺不可错
2. **代码自解释：** 代码能读出的"做了什么"不要再写文档
3. **补充 WHY：** "为什么这么做"、"踩过哪些坑"这类隐性知识必须写下来
4. **结构化索引：** 用 markdown 描述有哪些仓库、什么场景看哪些

为什么不推荐 RAG？
- RAG 中的数据更新无法保证
- 代码涉及多版本问题，RAG 不能解决
- topN 检索结果可能变成 AI 投毒源
- 如果某个知识点有价值，应人工记录下来放到知识库，而非依赖历史检索

### Prompt 优化：keep / cut / merge / move

四步循环，迭代 2-3 轮：

- **Keep：** 去掉就出错的，保留
- **Cut：** 不提示模型也能做好的、重复的、上级提示词已明确的，删除
- **Merge：** 意思相近的，合并
- **Move：** 同类 prompt 放一起，结构化呈现

目标是裁到原来的 1/3，保留原意。

### 强制回忆机制

每次唤醒 Agent 消费新消息时，强制注入几十个 token 让模型回忆最重要的原则。同样适用于监工 Agent：每次检查时强制回忆"什么算异常、何时告警"。

### Worklog 管理

- 每次问答记录，每天一个文件
- 跨天时压缩：保留有价值内容，合并冗余
- 核心知识点应进入知识库，而非依赖 worklog
- 不要让 Harness 自己管理 worklog——容易被投毒

## 多角色 Agent 协作

三个独立的 tmux session，各自独立上下文和 prompt：

### Oncall Agent（苦力）
- 职责：接收用户消息、回答技术问题、查知识库+代码、写 worklog
- Effort: 高

### 监工 Agent（Supervisor）
- 职责：监控所有 session、检测异常状态、判断是否该唤醒、告警通知人工、判断人工是否已介入
- Effort: 低
- 通过 `tmux capture-pane` 获取输出，通过 `tmux send-keys` 干预
- 状态判断是高风险操作，需要 AI 判断 + 明确规则兜底
- 发现人工已 attach 时绝不干预

### 代码分析 Agent
- 职责：分析复杂代码问题、追踪调用链、输出结论
- Effort: 高
- 独立运行，不污染 Oncall Agent 上下文
- 理想情况：Oncall Agent at 代码分析 Agent → 独立分析 → 返回结论 → Oncall Agent 继续服务用户

## 工程基础设施

### tmux 作为运行底座

- 每个角色对应固定 tmux session name
- 支持远程 ssh attach 介入
- 支持 `capture-pane` 监控 + `send-keys` 干预

### Harness 选型

- Bash 工具即足够（Agent 不需要改代码）
- Subagent 能力是加分项（避免复杂问题撑爆上下文）
- 不要自研 Harness——投入产出比太低
- 不要让 Harness 自己管理记忆

### Compaction 配置

- 上下文使用率超过 20% 后，质量下降很快
- Compaction 后有损总结会丢失关键信息
- 保守设 compaction 阈值（15-20%），宁可频繁 compaction 也不接近满载
- 接近满载时缺 Cache，Token 消耗飞快

## 监控与异常处理

### 三层防护

1. **监工 Agent 自动检测：** Agent 卡死、输出异常、消息积压、上下文水位。自动重试或通知。
2. **通知人工介入：** IM 推送告警，附带最近输出和积压情况。人工 ssh → tmux attach 介入。
3. **事后复盘：** Worklog 回顾 → 知识库补漏 → Prompt 修正。

### 关键检测规则

| 检测项 | 方式 | 阈值建议 |
|--------|------|---------|
| 卡死检测 | capture-pane 获取最后输出时间，纯规则 | 120-300 秒 |
| 循环检测 | AI 判断语义重复 | 最近 20 行 |
| 积压告警 | 消息目录文件数 | 按场景定 |
| 上下文水位 | Harness 状态输出 | 15-20% |

### 监工最小 Prompt

每 60 秒检查：capture-pane → 判断状态 → 正常则跳过 / 异常则告警 / 有人工 attach 则退出。

## 快速启动 checklist

1. [ ] 确定场景和首批知识库内容
2. [ ] 部署 tmux，创建 3 个 session
3. [ ] 配置 IM 消息订阅到目录
4. [ ] 配置 Harness，compaction 设 15-20%
5. [ ] 编写初始 CLAUDE.md / AGENTS.md
6. [ ] 编写外部消息脚本（订阅监控 + 检测 + 唤醒）
7. [ ] 编写监工 Agent 监控脚本
8. [ ] 上线 → 观察 → 迭代 prompt
