背景

过去多年，许多团队都尝试构建"AI" Bot 来解决 Oncall 中遇到的问题，
技术路线从关键词匹配、深度学习 + NLP，一路演进到 RAG，
但始终没能用好 Agent —— 之前的方案本质上都只是一问一答的 bot。

服务框架 Go 团队过去也在 Oncall 上投入了不少人力折腾，效果都不理想。
我曾多次建议切换到 AI Agent 路线，直接基于可信知识库 + 代码来回答，但当时并没有得到重视。

随着投入到 Go 框架的人力减少，Oncall 事务的占比越来越大，
我于是花了一天时间把 AI Oncall Agent 架构搭建了起来。
该 Agent 于 4 月初正式上线运行，逐步接管了人工 Oncall，目前每天仅剩零星一两例需要人工介入 —— 一部分是 AI 回答确有疏漏、需要补进知识库的，另一部分是用户坚持要找真人，不相信AI Agent可以解决问题。

本文主要分享如何快速构建一个 AI Oncall Agent，以及迭代过程中的经验，
也顺便回应那些对 AI Agent 持保留态度的人 —— 在 Oncall 这种场景下，认真做出来的 AI 确实可以比人更高效、更稳定。

本文不会分享或推荐任何代码，请不要做伸手党。
通用的代码并不值钱 —— 授人以鱼不如授人以渔，prompt 和架构思路才是核心。

旧架构问题

过去常以一问一答的方式工作，通过以webhook方式接到用户请求、查询RAG、总结回复。
用户多轮咨询时因为缺少上下文，会显得十分智障。
加之过度依赖低质量、过时无人维护、表达力差的 Wiki 文档，或者同样存在这些问题的 Oncall 历史记录，让智障雪上加霜。
最终只能解决一些很简单、很明显的问题，没能产生多少价值。
Go 框架团队交付的是代码，如果 AI Bot 不能结合最准确的代码上下文来回答，就毫无意义。

新架构介绍

最初 Larksuite CLI 还没有推出，我基于 Lark OpenAPI 自己搭了一套 CLI。
最核心的想法是：订阅 Lark 消息作为 Claude Code 的输入，在框架的仓库跑起来，再通过 CLI 发送或回复消息，并且不会关闭Claude Code、让其长驻，天然就拥有了上下文。

后来 Larksuite CLI 推出后，我才正式切换过去，开始真正把 AI Agent 搭建起来。
因为这时候刚好觉得团队应该需要改革Oncall了，正也好Lark有官方CLI。
于是通过 `lark-cli event +subscribe` 订阅消息到目录，
再通过 Claude Code 的 `/loop` 功能来消费这个目录 —— 就是这么简单，主要由 prompt 驱动。

这是最基础的架构，后面其实都是基于这样的架构演变。
当然这种方式也存在不少问题，后面章节会详细讨论。先把更重要的事分享完。

上下文优化

最开始只是为了方便我自己做代码分析和 Coding，所以建了个目录，把用到的仓库放进去，再执行一次 Claude Code 的 `/init` 就好。

后来发现没那么简单：我们的代码质量太低，不少用法连注释都没有，根本看不出是怎么回事，老是需要人来补充上下文。比如 Consul 不是那个 Consul，Mesh 是要配置而非透明接入的，懂的都懂。
所以要把 Agent 开放给外部用户，必须先建立一个高度可信、有明确 owner 的知识库，并确保其内容清晰、简洁、及时更新。

一个反面例子是 Hertz 框架 —— 曾经写了无数文档，甚至连许多和框架无关、只是 HTTP 相关的内容都写了一大堆，时间一长根本无法维护，反而给 AI 投毒。

我们为 Go 框架周边的 TCE / 服务发现 / Mesh 等基础知识都准备了一份说明，确保 AI Agent 在回答时能感知到这些组件的存在。知识库需要满足以下要求：

- 所有内容都必须经过人工校对、确认；
- 代码里能直接读出"做了什么"的部分，不要再单独写一份文档；
- 但"为什么这么做"、"踩过哪些坑"这类只有人才知道的上下文，必须通过文档写下来并且关联到代码；
  - 当然，更好的方式是直接在代码注释里能够交待清待。
- 可以把代码结构总结、索引为文档，加速 AI Agent 浏览代码。

更具体一点：我们把依赖到的所有代码仓库都 clone 下来，并在上层用 markdown 描述都有哪些仓库、什么场景下看哪些 —— TCE 相关的看哪些，服务发现的看哪些，Mesh 的看哪些。

当这些维护好之后，我们发现无论是不是 Oncall 场景，做方案设计时都能极大提效，因为 AI 已经掌握了"服务跑在 TCE 上、流量走 Mesh、流量治理由 Mesh 负责"这些背景知识。

后续我们还对用户经常问到的代码片段加强了注释，引入了 AI 来生成尽可能简洁的 changelog，以回答版本升级相关的问题。这些都是为了AI Agent可以尽快了解到相应的上下文。

本质上，这就是在不断地优化 context engineering，这是一个长期的事情。

除了基本的知识库上下文外，我们还引入 worklogs 上下文：每次回答用户问题时都会记录下来，每天一个文件，跨天时整理压缩，保留有价值的内容。这个也有帮助当我们切换 Harness 时，还能回顾一下当天的事情。

提示词优化

为了让 AI Agent 拥有人格，最早我甚至专门写了一份 whoami.md 文档，占了 1200 多 token，就为了让用户在 Oncall 时不那么枯燥。在 Opus 4.6 下效果的确不错，风格很像真人，甚至会反驳自己不是 AI 机器人。后来 Opus 4.7 推出后，不知是好是坏，风格变得非常机械，只管执行任务，人格设定和说话风格都消失了。

这时我就很能理解为什么用户总缅怀 ChatGPT 的 GPT-4o 或 o1 / o3 系列了。

关于 Opus 4.7 的这种变化，我觉得是一个很好的警示：如果我们的目标是解决用户问题，说话风格其实没那么重要，多余的提示词反而会增加模型对齐的难度。

刚开始上线、正式对接 Oncall 时，我几乎每天都在优化提示词。
因为凡是你没有明确说明的事情，模型都会自我发挥；有时候你都不知道某些知识点是从哪里学来的，必须追问才能发现那是幻觉还是知识库被投毒。

过了一阵子，一不留神就把提示词写到了 100k+ token，
这里面包括许多Oncall流程的指示，甚至包括了当AI Agent解决不了问题时，怎么转向内部群来咨询求助。

然而，发现随着运行时间变长、上下文变长，效果越来越差，开始出现各种不遵循 prompt 的 AI 幻觉。
最初很难取舍，总觉得每条 prompt 都是必要的，只能折中 —— 把 prompt 拆成多个 markdown 文件管理。

但即使拆开后还是有不少问题，比如 AI 以为自己读过、理解了，可一旦上下文一长就又不行了。
特别是像 Opus 这种声称支持 1M Context 的模型，在 Oncall 这种多用户、多上下文混杂的场景下，上下文超过 20% 之后质量下降很快。反倒是后面换成 GPT 5.5 后，虽然上下文窗口更小，效果却更好，而且刚好 Trae CLI 也支持 GPT5.5了，很好很合规。

这里还发现一个问题：发生 compaction 后，上下文中主要保留的是 CLAUDE.md/AGENTS.md 的内容，其余内容虽然可能保留，但都是被总结过的、有损的。

所以最终还是必须把提示词精简下来，尤其是 CLAUDE.md/AGENTS.md 及其引用内容这两级。
我通过以下方法把提示词规模裁到了原来的 1/3，并且保留了原意 —— 总结起来就是 keep / cut / merge / move 原则：

- keep：哪些是必要的、不能再精简的，确保新文件里保留；
- cut：删掉那些你不提示、顶级模型也能做好的，或重复的、上级提示词里已经明确过的；
- merge：把两条或多条意思相近的 prompt 合并，不要啰嗦；
- move：把同类 prompt 放在一起，结构化呈现，让模型更容易理解，也方便后续做 cut / merge。
  按上面的规则反复迭代几次后，prompt 就十分精简了。我还把整个流程整理成了 skill，方便复用。

持续优化一两周后，效果也趋于稳定，但模型不遵循 prompt 的问题仍然存在。
没办法，我最终在消费消息的提示词里强制让模型回忆一下最重要的几点 ——
你可以理解为：每当有新消息到来通知模型消费时，都强制注入几十个 token 告诉它"回忆一下 XX、XX 最重要"
这样它就永远不会忘记最重要的原则了。

关于 Harness

在 AI Oncall Agent 里，其实 Harness 那一层并没有那么重要。我甚至专门写了提示词，让 Harness 不要自己瞎记东西 —— 一旦被投毒，你都不会知道。

有一天我自己实现了一个极简的 Harness，想用来跑 Oncall Agent，觉得专用的 harness 应该会更好。
但发现在当前场景下，唯一的好处是：哪怕用我 Macbook 上的本地模型也能跑得很好，因为它确实非常节省提示词，并且开箱即用。

极简的 Harness 是可行的 —— 只要有 Bash 这一个工具，Harness 几乎可以干任何事情，我们也不需要 Oncall Agent 改代码。
但你会发现 AI 经常会很努力地把多条指令用管道拼起来，就为了干一件很简单的事，中间又可能因为某个命令不存在、或 GNU 工具语法差异而失败，反而浪费时间。
更重要的是交互和用户体验方面，现有的通用 Harness 已经做得足够好，你可以清晰看到上下文占用和当前正在执行的任务。

所以最终我还是决定短期内放弃自研 Harness 这条路。在最新架构下，我们通过 tmux 来运行 Trae CLI / Codex CLI / Claude Code 这些工具：
如果 Agent 在执行一个很重的任务、可能会卡很久，这时我们可以随时登录 attach 上去中断它，或者在出错时给出指令重试。

tmux 的监控也可以交给 AI：通过 `tmux capture-pane` 获取输出，通过 `tmux send-keys` 进行干预。
所以基本上你需要的只是一个苦力 Agent 加一个监工 Agent，二者架构完全一样、一起部署，只是提示词不同。
监工 Agent 本身几乎没什么提示词，理论上不用大模型也能胜任 —— 但要注意它判断的"是否忙碌 / 是否该唤醒 / 是否人工已介入"是高风险的状态判断，误判会导致打断人工或漏掉用户消息，所以如果用小模型，需要配合明确的规则兜底。

我觉得唯一需要顾虑的，是 Harness 是否具备 subagent 能力 —— 你不希望用户问一个问题就把当前上下文占满，尤其是分析复杂代码问题时。但问题是，哪怕 Harness 支持 subagent，模型也不一定会用。

无论你用什么样的Harness，设置Compaction的比例十分重要。
特别是 Opus 这种1M Context的，Context快满时如果缺少Cache会消耗Token飞快，而且每次的响应都变慢许多。

工程优化

分享完上面这些后，再说一下工程方面的问题。
最早并没有用 tmux，总是在本地开着窗口运行。
时间久了发现这样不便于随时随地接入干预，特别是远程ssh的情况，于是引入了 tmux。
用一段时间后发现 tmux 挺好用的，通过 tmux 的命令行还能再挂一层 AI 做监控，于是就一直沿用下来。

中间也测试过用 `claude -p` / `codex exec` 换成单次运行的方式，但考虑到每次都要恢复上下文，加上人工干预体验也不如 tmux，最终还是放弃了。

前面提到，最早是通过 `lark-cli event +subscribe` 订阅消息到目录，再通过 Claude Code 的 `/loop` 来消费。但发现长时间没有消息时，`/loop` 会十分消耗token和上下文，而且容易把已有的上下文挤掉。

本质上，我们需要把消息通知这个逻辑从 Harness 里抽出来，形成"订阅 → 通知 → 检测 → 唤醒 → 批量处理"这样的流程：

- 订阅 → 通知：直接监控订阅目录即可；
- 通知 → 检测：主要判断 Agent 是否处于忙碌状态，或是否有人工 attach 到 tmux 上处理问题 —— 这一步我们尝试用 AI 判断告警，效果不错；
- 检测 → 唤醒：使用预设的唤醒提示词，这一步很重要 ，你可以每次都重复一遍最重要的提示；
- 唤醒 → 批量处理：解决多条消息到来时上下文相关的处理问题，不应该逐条孤立处理。

这样抽象之后，Harness 用什么其实都无所谓，只要把"订阅 → 通知"这一段抽出来由外部处理，然后通过 `tmux send-keys` 唤醒就行。
但消息怎么消费和归档，还是需要写一点脚本让 AI 按规范执行，而不是直接读目录里的 JSON 再 rm 掉 —— 换个角度看，这样也更节省 token。
不管是用 Trae CLI 、 Claude Code 还是 Codex，只要确保每个角色对应统一的 tmux session name 就行。

当有了一个 Oncall Agent 之后，大家总会想：既然都有 Agent 了，那就什么事都丢给它办吧。
但别忘了，这样会堵塞 Oncall Agent 的工作流，还会占用本不相关的上下文。这时候就需要引入多角色来分担 —— 让 Oncall Agent 继续专注 Oncall 本身。在搭建 Oncall Agent 时，我同步建立了多个角色，包括前面提到的监工 Agent，以及一个没有提及的、专门分析代码的 Agent。

在试运行了一段时间，发现这样的效果不错，各司其职，我还能为不同的Agent设置不同的effort level。因为监工Agent的存在，我几乎能发现一切的异常，只需要你写一句prompt让发现异常汇报给你。

我于是进一步想，他们是否可以闭环起来。

我建了一个群，把几个 Agent 都拉进去，希望让它们互相沟通。比如说，当Oncall Agent需要分析代码时，就在群里把任务交给另一个Agent来执行，Oncall Agent继续服务其他用户。

但由于 Lark 设计上有不少缺陷，一个 Agent 是无法接收到另一个 Agent 的消息的，哪怕被 at 了也收不到 —— 你只能通过脚本列出最近的消息，看有没有别的 Agent 发消息，然后再通过线下通道通知另一个 Agent。
后来 Lark 可能发现这个BUG了，于是额外引入了一个叫`im:message.group_at_msg.include_bot:readonly` 权限，才解决问题。

然而事情还没完：当前如果一个 bot 想 at 另一个 bot，并没有优雅的办法 —— 因为 bot 没有 user id，另一个 bot 也无法取到对端 bot 的 open id，对于大部分接口只有一个 app_id，而这个app_id绝大部分接口都用不上，想真正 at 成功并不容易。
估计 Lark 内部就没有真正实践过多 AI Agent 协同 —— 至少管理层没做过吧。

所以在多 AI Agent 架构这块的优化一直被 Lark的问题拖着，只能要么人工 at，要么给每个 Agent hack 一下，拿到每个对端 Agent 的 open id 并记下来（注意 open id 是不能共享的），不好扩展起来。

总结

我们之所以这么做、这么优化，是因为这是 Oncall 场景，而我们是框架团队，主要交付代码 —— 每天可能就那么几十个用户过来，AI 效率不是最重要的，内容准确才是。
所以我们的方案不一定适用于你的场景，仅供参考。

整个过程中我们并没有用到 RAG，因为光是Wiki我们都维护不好，凭什么RAG里的数据你就能确保更新? 而且像代码之类还涉及到多版本问题，RAG并不能解决什么。
在实际 Oncall 中，真正能通过 RAG 命中并直接回复用户的情况本来就不多；如果再叠加没人保证内容正确性的问题，topN 检索结果只会变成对 AI 的投毒源。
特别是通过历史Oncall数据来建立知识库检索这事本身就没有多少意义，如果某个知识点有价值，我们应该是人工把它记录下来，而不是依赖历史去回复将来的问题。
所以我们选择直接基于可信知识库 + 代码来回答 —— 这本质上也是一种检索，只是把"什么时候读哪些文件"交给了人工编排和 prompt 控制，可控性更强。

当前架构可能略显简陋，毕竟投入时间也不多，但至少有效 —— ROI 可以说是团队成立以来最高的，比 Oncall 平台那些搞了几年的项目都好。

如果你也想搭建一套 AI Oncall Agent，并且是面向外部用户的，我并不建议照搬上面的方法。这里有几点建议：

- Lark 订阅消息通过 webhook 接收，按用户做 sharding 后投递到 MQ；
- 消费 MQ 处理这些消息，确保有足够实例打散不同用户的上下文；
- 通过 Hook 实时把 Harness 的上下文保存到数据库，形成工单，并实现 MCP 让 AI 可以加载工单上下文；
- 通过定时任务，定期整理数据库里的工单内容，形成优化建议，闭环优化产品；
- 尽可能按职责细分、拆分Agent，这样每个Agent的上下文、提示词都更少。
