监控与异常处理

监工 Agent 兜底 + 人工介入入口 — 确保系统不会静默挂掉

三层防护体系

从自动检测到人工介入的逐级响应
  第 1 层：监工 Agent 自动检测
  ┌─────────────────────────────────────────┐
  │ · Agent 卡死（超过 X 秒无输出）            │
  │ · Agent 输出异常（重复循环、格式错误）       │
  │ · 消息积压（目录中未消费消息超过阈值）        │
  │ · 上下文即将满载（compaction 未正常触发）    │
  │                                          │
  │ 动作：自动 send-keys 重试 / 通知你          │
  └──────────────────┬──────────────────────┘
                     │ 监工无法自动恢复
                     ▼
  第 2 层：通知你介入
  ┌─────────────────────────────────────────┐
  │ · IM 推送告警消息（"Agent 卡死了"）         │
  │ · 附带最近 N 行输出 + 积压消息数            │
  │ · 你可以直接 ssh → tmux attach 查看        │
  └──────────────────┬──────────────────────┘
                     │ 需要深入排查
                     ▼
  第 3 层：事后复盘
  ┌─────────────────────────────────────────┐
  │ · Worklog 回顾：哪条消息触发了异常？        │
  │ · 知识库补漏：是否因为缺少某个知识点？       │
  │ · Prompt 修正：是否提示词未覆盖这种边界？    │
  └─────────────────────────────────────────┘

关键检测规则

卡死检测

规则：上次输出距今 > X 秒

capture-pane 获取最后输出时间
纯规则判断，不需要 AI
推荐阈值：120-300 秒

循环检测

规则：最近 N 行输出重复模式

用 AI 判断语义重复
比纯文本匹配更可靠
发现→强制中断+重试

积压告警

规则：消息目录文件数 > 阈值

说明 Agent 处理速度跟不上
可能是卡死，也可能是负载太高
需要区分处理

上下文水位

规则：Context 使用率 > 阈值

从 Harness 的状态输出获取
接近满载前主动告警
预防性触发 compaction

监工 Agent 的最小 Prompt

极简但精确——监工不需要大模型能力
每 60 秒执行以下检查：

1. 对每个被监控的 tmux session 执行 capture-pane
2. 判断：
   a. 最后输出的时间戳——超过 180 秒未更新 → 可能卡死
   b. 最近 20 行是否有重复模式 → 可能在循环
   c. 消息目录是否有新文件堆积 → 积压
3. 决策：
   - 如果看起来正常（最近有输出、无重复）→ 什么都不做
   - 如果 Agent 空闲且消息积压 → send-keys 唤醒
   - 如果 Agent 卡死或循环 → 通知 [你的 IM]
   - 如果你的判断不确定 → 通知 [你的 IM]，让人类决定
4. 重要：如果通过 capture-pane 看到人工正在输入（attach），
   绝对不要 send-keys 干预——退出检查即可
