工程基础设施

tmux 托管 + Harness 选型 + Compaction 策略 — 让 Agent 稳定长驻运行

tmux 作为运行底座

为什么用 tmux 而不是直接跑进程？
需求 直接跑进程 tmux 托管
远程介入 需要重新连接 ssh 上去直接 attach
查看实时输出 要看日志 capture-pane 即时获取
中断/重试 kill 后重启 attach 后 Ctrl+C 或 send-keys
AI 监控 需要额外适配 capture-pane / send-keys 天然适配

关键约定：每个角色对应固定 tmux session name，监工才能精准监控和唤醒

tmux 部署架构

一台机器上的 tmux 布局
  tmux ls
  ┌─────────────────────────────────────────────────────┐
  │ session: oncall-agent    │  Claude Code /loop       │
  │                          │  消费消息目录             │
  │                          │  Effort: high            │
  ├─────────────────────────────────────────────────────┤
  │ session: supervisor      │  轻量循环脚本              │
  │                          │  capture-pane 监控        │
  │                          │  send-keys 唤醒           │
  │                          │  Effort: low             │
  ├─────────────────────────────────────────────────────┤
  │ session: code-analyzer   │  按需启动                 │
  │                          │  被 at 后分析代码          │
  │                          │  Effort: high            │
  └─────────────────────────────────────────────────────┘

Harness 选型：不重要但需注意

选择标准

  • 核心能力：Bash 工具即可（Agent 不需要改代码）
  • Subagent 能力：有助于避免复杂问题撑爆上下文，但模型不一定会用
  • 上下文可见性：能否清晰看到 token 占用和当前任务状态
  • 模型兼容：Claude Code / Trae CLI / Codex CLI 都可以，按合规和效果选

不要做的事

  • 不要为了 Oncall Agent 自研 Harness——投入产出比太低
  • 不要让 Harness 自己管理记忆——被投毒了你都不知道
  • 不要频繁切换 Harness——稳定运行 > 功能炫酷

Compaction 配置（重要）

决定上下文质量和 Token 成本的关键参数

原则

  • 上下文使用率超过 20% 后，1M Context 模型质量也下降很快
  • Compaction 后保留的主要是 CLAUDE.md/AGENTS.md 内容，其余都是有损总结
  • 宁可频繁 compaction 也不让上下文接近满载——满载时缺 Cache，Token 消耗飞快

操作建议

  • 设一个保守的 compaction 阈值（如 15-20%）
  • 定期检查 compaction 后的上下文质量
  • 如发现 compaction 后丢关键信息，说明 prompt 需要更精简
