上下文工程：知识库 + 提示词 + Worklog

决定 AI Agent 回答质量的核心不是模型，而是它"看到了什么"

三层上下文架构

Agent 的上下文金字塔
                    ┌─────────────────┐
                    │   Prompt 层      │  ← 最小、最核心、强制回忆
                    │   (CLAUDE.md等)  │
                    ├─────────────────┤
                    │                  │
                    │   知识库层       │  ← 可信文档 + 代码索引
                    │   (按需加载)     │
                    │                  │
                    ├─────────────────┤
                    │                  │
                    │   代码仓库层     │  ← 直接读源码，不通过 RAG
                    │   (按需浏览)     │
                    │                  │
                    ├─────────────────┤
                    │                  │
                    │   Worklog 层     │  ← 每日记录，跨天压缩
                    │   (历史上下文)    │
                    │                  │
                    └─────────────────┘

原则：Prompt 越少越好，知识库越可信越好，代码直接读，Worklog 及时压缩 —— 防止上下文膨胀导致质量塌陷

知识库设计原则

1. 可信优先

所有内容必须人工校对确认。
宁可缺，不可错。
错误的知识库是在给 AI 投毒。

2. 代码自解释

代码能读出的"做了什么"，
不要再写文档。
减少维护负担，避免文档腐化。

3. 补充 WHY

"为什么这么做"、"踩过哪些坑"
这类只有人知道的隐性知识
必须在文档或注释里写下来。

4. 结构化索引

用 markdown 描述：有哪些仓库、
什么场景看哪些。
加速 AI 定位正确的代码位置。

Prompt 优化：keep / cut / merge / move

精简前 (100k+ token)

症状：

  • 上下文一长就开始不遵循 prompt
  • compaction 后有损总结丢失关键信息
  • 不知道哪些 prompt 真正有用
精简后 (原 1/3)

四步循环：

  1. Keep: 去掉就出错的，保留
  2. Cut: 不提示模型也能做好的，删
  3. Merge: 意思相近的，合并
  4. Move: 同类放一起，结构化

反复迭代 2-3 轮，效果就稳定了

强制回忆机制（关键技巧）

防止模型遗忘核心原则的最后防线

每次唤醒 Agent 消费新消息时，强制注入几十个 token：

"在回复用户之前，先回忆以下最重要的几点：
 1. 所有回答必须基于可信知识库和代码——禁止编造
 2. 如果答案不在知识库里，明确告知用户并记录下来
 3. 不要猜测代码行为——直接读源码确认
 4. ..."

同理也适用于监工 Agent：每次检查时强制回忆"什么算异常、何时告警"，确保行为一致性

Worklog 管理

记录策略

  • 每次问答都记录，每天一个文件
  • 跨天时压缩：保留有价值内容，合并冗余
  • 有 AI 自动压缩 + 人工抽查关键条目

注意

  • 不要让 Harness 自己管理 worklog——容易被投毒
  • worklog 是给 Agent 回顾用的，不是给 RAG 检索的
  • 压缩是有损的，核心知识点应进入知识库而非依赖 worklog
