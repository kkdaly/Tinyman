监工 Agent: 关键设计

监工的工作循环
每 N 秒循环:
  capture-pane → 看输出
  ┌─ 是否忙碌中？
  │   是 → 跳过，等下一轮
  │   否 → 有新消息？
  │        是 → send-keys 唤醒
  │              Oncall Agent
  │        否 → 继续等
  └─ 异常？
       是 → 通知你

监工 prompt 极短，几乎不需要大模型
但状态判断是高风险操作，需要规则兜底

监工判断的风险点
  • 误判"空闲" → 打断正在处理的任务
  • 误判"忙碌" → 用户消息被晾着
  • 误判"人工未介入" → send-keys 干扰人工操作
  • 误判"无异常" → Agent 卡死无人知晓

对策：小模型 AI 判断 + 明确规则（如：上次输出超过 X 秒未更新=卡死）

代码分析 Agent: 按需调用

为什么需要独立的代码分析角色？

当 Oncall Agent 需要深入分析代码时，如果不拆分：

  • 分析代码消耗大量上下文，挤掉前面用户的对话
  • 分析期间 Oncall Agent 被阻塞，其他用户消息无法处理
  • 代码分析的上下文和 Oncall 上下文混在一起，互相污染

拆分后：Oncall Agent 在群里 at 代码分析 Agent → 代码分析 Agent 独立分析 → 返回结论 → Oncall Agent 继续服务用户

注意：这也依赖 IM 平台的 bot 间通信能力（Lark 尚不完善），替代方案是人工 at 或脚本中转
